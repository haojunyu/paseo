# 编码规范

核心直觉：AI 生成的代码在规避——它覆盖所有情况，层层叠加而非切入核心，把不确定性散布各处，用包装来兜底。而资深工程师做的是承诺——承诺一个形态、一条边界、一个命名、一条正常路径、一个类型——然后让其他一切都自然归位。以下每一条规则都是在捕捉不同形式的犹豫不决。

测试规则请参见 [testing.md](testing.md)。

## 核心原则

- **零复杂度预算** —— 每个抽象都必须以明确、当下的收益来为自己辩护。
- **YAGNI** —— 只在需要时才构建功能和抽象。一个只被调用一次的函数是间接层，不是抽象。
- **禁止"顺手"清理** —— 只做你来的目的所需的改动。附带修改会隐藏在 diff 中。
- **函数式和声明式** 优先于面向对象。
- **`function` 声明** 优先于箭头函数赋值。
- **`interface`** 优先于 `type`，当两者都适用时。
- **禁止 `index.ts` 桶文件** 只做重新导出——它们制造间接层和循环依赖风险。直接从源文件导入。

## 注释和噪音

- 删除任何删除后不损失任何信息的注释。注释解释_为什么_，而非_是什么_。
- 禁止教学式注释解释语言特性（`// 使用解构来…`）。
- 禁止装饰性分隔符（`// ===== 辅助函数 =====`）。用文件和模块来组织，而非 ASCII 艺术。
- 禁止犹豫性注释（`// 可能需要重新审视`、`// 大多数情况下应该能用`）。如果不确定，就去搞清楚。
- 禁止被注释掉的代码。Git 会记住。
- 禁止残留的 `console.log` / `debugger`。禁止 `TODO: implement` 占位桩——如果需要存在，就把它写出来。

## 自信：承诺一个形态

- 在边界处验证（网络、IPC、用户输入、文件 I/O），在类型系统内部信任类型。经过解析后，值就是其类型所声明的样子。
- 验证边界之后的每一个 `?.` 和 `??` 都是不自信的代码——要么边界应该解决它，要么类型应该反映现实。
- 禁止对类型系统已经排除的条件做防御性检查（对非空参数做 `if (!agent) return`）。
- 禁止"以防万一"的 `try/catch`。如果你说不出你在捕获什么以及为什么，就不要捕获。
- 可选性是一个设计决策，不是迁移捷径。明确的不同有效状态 → 可辨识联合类型。有意为空 → 明确的 `null`。让可选性留在真正的边界处。

## 类型

- 禁止 `any`。禁止用 `as` 强制转换来绕过错误。禁止 `@ts-ignore` / `@ts-expect-error`。用 `if` / schema 验证来收窄；让编译器检查得更严，而不是更松。
- 如果存在 Zod schema，那么 TypeScript 类型就是 `z.infer<typeof schema>`。绝不手写一个并行的类型。
- 每个概念只有一个权威类型。层级特定的视图用 `Pick` / `Omit`，而非复制字段。
- 为多属性的对象形态命名——禁止在签名、返回值或泛型参数中出现内联的 `Array<{ ... }>` 或 `Promise<{ ... }>`。
- 当值是已知集合之一时，使用字符串字面量联合类型，而非原始 `string`。在编译时捕获拼写错误。
- 超过显式命名阈值的对象参数：3+ 个参数、任何布尔参数、任何可选参数 → 使用对象。`(thing, true, false, true)` 在调用处是不可读的。
- 让不可能的状态变得不可能——用可辨识联合类型替代 `{ isLoading; error?; data? }` 这样的杂糅体。

## 错误

- 抛出带类型的错误类，携带调用方想要读取的字段。普通的 `Error("Provider X not found")` 把结构化信息折叠成了字符串。
- Catch 块用 `instanceof` 分支处理它们能处理的；其余重新抛出。禁止 `catch (e) { return null }`。
- 将面向用户的文案与日志/调试字符串分离——不要让一个字符串同时服务于遥测、日志和 UI。
- 明确失败。如果调用方请求了 X 而 X 不可用，抛出异常——不要默默地替换成 Y。

## 密度

- 嵌套三元表达式被禁止。单个三元表达式仅在两个分支都是单个标识符或简单访问（`x ? a : b`）时可以接受。
- 包含 2 个或更多子句或混合关注点的布尔表达式 → 为条件命名。
- 对象字面量组装预计算的值；不要把分支和查找塞进属性位置。
- 操作嵌套操作（`Object.fromEntries(arr.filter(...).map(...))`、`Math.max(...xs.map(...))`）→ 拆分为命名的中间变量。
- 最多 3 层嵌套（回调、JSX、控制流）。超过则提取。

## 结构和模块

- 目录是一个模块，不是一个命名空间。一个有意的公共接口面；内部文件保持内部。
- 路径是名称的一部分——优先用 `provider/registry.ts` 而非 `provider/provider-registry.ts`。如果文件名需要承担双重职责，就加深路径。
- 以 `-utils`、`-helpers`、`-manager`、`-handler`、`-controller`、`-formatter`、`-builder` 结尾的文件名是一种坏味道——路径没有承载足够的领域含义。
- 边界返回值回答调用方的问题（`getActiveAgents()`），而非"这是我的存储"（到处重复 `getAgents().filter(...)`）。
- 一个适配器意味着一个假设的接缝；两个适配器意味着一个真实的接缝。在某样东西真正跨它变化之前，不要定义端口。
- 透传模块无法通过删除测试——如果删除该模块后调用方直接得到它们想要的，就删除它。
- 集中化策略。同一个鉴别器（`plan`、`provider`、`kind`、`status`）在 3 个以上文件中分支 → 策略表，而非每个 case 又一个 `else if`。
- 新功能在实现之前先有一个归宿。一个功能散落在 5 个共享文件中，和平铺的命名空间一样混乱。
- 不要仅因为放置位置不明确就把新文件丢在最近的根目录下——说出来并询问。

## 重构是一次附加测试

- 一个改动应该看起来像对现有代码的一次深思熟虑的编辑，而不是在它旁边加了一层新东西。新的协调器包裹协调器、新标志绕过正常路径、新辅助函数复制已有的选择器——停下来，重新塑造。
- 重构默认保持行为不变。未经明确批准，不得为了简化代码而删除功能。
- 在重构_之前_制定验证计划——命名不变量，确认有测试覆盖它们，如果没有就写一个。参见 [testing.md](testing.md)。
- 在同一次重构中迁移所有调用方并删除旧路径。除非明确设计，否则不留回退行为。

## React

- `useEffect` 用于与外部系统同步（DOM、网络、定时器、订阅）。不用于转换 React 状态。派生状态 → 在渲染中计算或使用 `useMemo`。
- 禁止 effect 级联——设置 state 触发更多 effect 的链条几乎总是应该用 React Query 或 reducer。
- `useRef` 用于 DOM 引用和非渲染身份（定时器 ID、AbortController、最新回调缓存）。如果值影响下一次渲染的内容，它就是状态——用 `useReducer` 和可辨识联合类型显式建模。
- 服务端状态通过 React Query 处理。手动 `useState` + `useEffect` + `isLoading` + `error` 来处理获取的数据总是更差。
- 组件负责渲染和派发——它们不计算状态转换。两个或更多相互作用的 `useState` → 提取为 reducer。
- 绝不在组件内部定义组件。仅限模块作用域。
- 窄订阅：从 store 中选择原始值，传递 `status` 而非 `agent`，在返回派生数组/对象时使用 `useShallow` / deep-equal。
- 跨越 `memo` 边界或作为依赖数组输入的 props 要保持引用稳定。模块作用域的静态字面量用 `as const`；派生值用 `useMemo`；处理器仅在存在记忆化受益方时用 `useCallback`。
- `key` 使用稳定的 id，对于可重排/可过滤列表绝不使用数组索引。
- Context 用于稳定值（主题、认证）。Store 配合选择器用于会变化的状态。

## 命名

- 命名描述含义，而非机制。`submitForm` 而非 `handleOnClickButtonSubmit`。`running` 而非 `filteredArrayOfRunningAgents`。
- 正确的长度是在上下文中最短且无歧义的。在 `AgentManager` 内部，方法是 `start`、`stop`、`list`。
- 匹配周围代码的词汇。如果代码库使用 `getX`，不要为同样形态引入 `fetchX` / `retrieveX`。
- 不要让实现细节泄漏到命名中——`getAgent`，而非 `queryPostgresForAgent`。如果替换实现会强制改名，那这个名字就是错的。
- 布尔值读起来像是/否问题：`isX`、`hasX`、`canX`。避免否定布尔值（`isNotConnected`）。
- `data`、`result`、`info`、`manager`、`temp` 是坏味道——说出这个东西_是什么_。
